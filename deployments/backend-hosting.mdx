The Mosaic Reporting backend API is packaged as a Docker image and run on a managed container platform — either AWS ECS Fargate or Kubernetes (EKS/GKE). Containerizing the API ensures that every environment from local development to production runs an identical runtime, eliminating dependency drift between machines and making rollbacks straightforward.

## Dockerfile

The backend uses a **multi-stage Docker build** to keep the final image small and free of development tooling.

```dockerfile
# ── Stage 1: Build ─────────────────────────────────────────────────────────────
FROM node:20-alpine AS builder

WORKDIR /app

# Install dependencies (including devDependencies for tsc)
COPY package.json pnpm-lock.yaml ./
RUN corepack enable && pnpm install --frozen-lockfile

# Copy source and compile TypeScript
COPY tsconfig.json ./
COPY src/ ./src/
COPY prisma/ ./prisma/
RUN pnpm build              # runs tsc → dist/
RUN pnpm prisma generate    # generate Prisma client into node_modules

# ── Stage 2: Production image ──────────────────────────────────────────────────
FROM node:20-alpine AS production

ENV NODE_ENV=production
WORKDIR /app

# Install only production dependencies
COPY package.json pnpm-lock.yaml ./
RUN corepack enable && pnpm install --frozen-lockfile --prod

# Copy compiled output and Prisma client from builder
COPY --from=builder /app/dist ./dist
COPY --from=builder /app/node_modules/.prisma ./node_modules/.prisma
COPY prisma/ ./prisma/

# Non-root user for security
RUN addgroup -S mosaic && adduser -S mosaic -G mosaic
USER mosaic

EXPOSE 3000

HEALTHCHECK --interval=30s --timeout=5s --start-period=15s --retries=3 \
  CMD wget -qO- http://localhost:3000/health || exit 1

CMD ["node", "dist/server.js"]
```

Key points:
- The `builder` stage installs all dependencies (including `devDependencies`) to compile TypeScript and generate the Prisma client.
- The final `production` stage installs only production dependencies and copies the compiled `dist/` from the builder, keeping the image lean.
- The container runs as a non-root user (`mosaic`) to limit the blast radius of any compromise.
- The built-in `HEALTHCHECK` instruction allows Docker and ECS to detect unhealthy containers during startup.

## Container Registry: Amazon ECR

Every CI build pushes the Docker image to an Amazon ECR private registry. Images are tagged with the full Git SHA:

```text
123456789012.dkr.ecr.us-east-1.amazonaws.com/mosaic-api:a1b2c3d4
```

ECR lifecycle policies automatically expire images older than 30 days that are not tagged as `latest` or referenced by a live task definition, preventing unbounded registry growth.

    kind: Deployment

      name: mosaic-api
      namespace: production
    spec:
      replicas: 2
      selector:
        matchLabels:
          app: mosaic-api
      strategy:
        type: RollingUpdate
        rollingUpdate:
          maxSurge: 1
          maxUnavailable: 0
      template:
        metadata:
          labels:
            app: mosaic-api
        spec:
          containers:

              ports:
                - containerPort: 3000
              env:
                - name: NODE_ENV
                  value: production
              envFrom:
                - secretRef:
                    name: mosaic-api-secrets
              resources:
                requests:

                  memory: "512Mi"

                  cpu: "500m"
                  memory: "1Gi"
              livenessProbe:
                httpGet:

                  path: /health

## Resource Limits by Environment

| Environment | CPU (vCPU) | Memory | Min Replicas | Max Replicas |
|-------------|-----------|--------|-------------|-------------|
| `development` | 0.25 | 512 MB | 1 | 1 |
| `staging` | 0.5 | 1 GB | 1 | 2 |
| `production` | 1.0 | 2 GB | 2 | 10 |

Auto-scaling in production triggers a new container launch when average CPU utilization exceeds 70% for two consecutive minutes. Scale-in occurs when CPU drops below 30% for ten minutes.

## Load Balancer

The AWS Application Load Balancer (ALB) sits in front of the ECS service (or Kubernetes Ingress controller):

- **Protocol**: HTTPS on port 443; HTTP on port 80 is redirected to HTTPS automatically.
- **TLS certificate**: provisioned and renewed automatically by AWS Certificate Manager (ACM).
- **Target group**: ECS tasks on port 3000; health check path is `GET /health`, interval 30 seconds, healthy threshold 2, unhealthy threshold 3.
- **Stickiness**: disabled — the API is stateless; any replica can serve any request.

## Health Check Endpoint

The backend exposes a lightweight health check endpoint used by the ALB and container health checks:

```http
GET /health
```

```json
{
  "status": "ok",
  "version": "1.4.2",
  "uptime": 38241,
  "database": "connected",
  "redis": "connected"
}
```

The endpoint returns HTTP `200` when the service is healthy. It returns HTTP `503` if the database or Redis connection is down, causing the ALB to route traffic away from that container until it recovers.

## Rolling Deployments

When the CI pipeline deploys a new image, ECS performs a **rolling update**:

1. ECS launches new tasks with the updated image alongside existing tasks.
2. The ALB's health check confirms new tasks are healthy before registering them as targets.
3. Old tasks are deregistered from the ALB and drained (existing connections are allowed to complete) for up to 30 seconds.
4. Old tasks are then stopped.

This process ensures zero downtime — at least one healthy replica is serving traffic at all times during a deployment.
