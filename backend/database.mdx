Mosaic Reporting uses **PostgreSQL 15** as its primary relational database, accessed exclusively through **Prisma ORM**. Redis complements PostgreSQL by handling ephemeral data — session tokens, job queues, and worklist caches — where low-latency key-value access is more appropriate than a relational query.

## Primary Database: PostgreSQL 15

All persistent application data lives in PostgreSQL. You interact with the database exclusively through Prisma — never through raw SQL in application code or ad-hoc migration scripts. PgBouncer sits in front of PostgreSQL to manage connection pooling, ensuring that the application never exhausts the database's connection limit under concurrent load.

## Redis Usage

Redis handles three distinct responsibilities in Mosaic Reporting:

| Responsibility | Description |
|---|---|
| **Session / token storage** | Access and refresh tokens are stored under predictable key patterns for fast validation on every request. |
| **BullMQ job queues** | Background jobs (HL7 dispatch, FHIR publish, PDF generation) are queued and processed using BullMQ, which uses Redis as its backing store. |
| **Worklist cache** | Filtered worklist query results are cached in Redis to avoid repeated expensive PostgreSQL queries on frequently accessed filter combinations. |

### Redis Key Patterns

| Key Pattern | Purpose |
|---|---|
| `session:<userId>` | Active session data for a given user |
| `refresh:<token>` | Refresh token payload, with TTL matching token expiry |
| `worklist:cache:<hash>` | Cached worklist result for a specific filter hash |
| `bull:<queue>:*` | BullMQ internal keys for job state tracking |

## Core Tables

The following tables form the core of the Mosaic Reporting schema. Each maps 1:1 to a Prisma model.

| Table | Description |
|---|---|
| `users` | Registered users of the platform, including radiologists, technologists, and administrators. Stores credentials, role, and profile information. |
| `reports` | The primary report record, linking a study to an assigned radiologist and capturing the current status and body content. |
| `report_versions` | Immutable version history for every report. Each save or amendment creates a new version row, preserving the full body at that point in time. |
| `worklist_studies` | Studies ingested from PACS, including DICOM metadata (modality, accession number, study instance UID), priority, and claim status. |
| `templates` | Report templates grouped by modality and body part. Templates provide a structured starting body for new reports. |
| `notifications` | In-app notifications generated by workflow events (e.g., a report assigned, a study marked urgent). |
| `audit_logs` | Immutable audit trail for security-sensitive and workflow events: logins, report finalization, amendments, HL7/FHIR dispatch outcomes. |

## Prisma ORM Configuration

Prisma models map 1:1 to database tables. Enums are defined in the Prisma schema and kept in sync with the database using `prisma migrate`. The three key enums are:

- **`UserRole`** — `ADMIN`, `RADIOLOGIST`, `TECHNOLOGIST`, `VIEWER`
- **`ReportStatus`** — `DRAFT`, `PENDING_REVIEW`, `FINAL`, `AMENDED`
- **`StudyPriority`** — `ROUTINE`, `URGENT`, `STAT`

### Simplified Prisma Schema

The following is a representative excerpt of the Prisma schema covering the three most central models. The full schema lives at `prisma/schema.prisma`.

```prisma
// prisma/schema.prisma

generator client {
  provider = "prisma-client-js"
}

datasource db {
  provider = "postgresql"
  url      = env("DATABASE_URL")
}

enum UserRole {
  ADMIN
  RADIOLOGIST
  TECHNOLOGIST
  VIEWER
}

enum ReportStatus {
  DRAFT
  PENDING_REVIEW
  FINAL
  AMENDED
}

enum StudyPriority {
  ROUTINE
  URGENT
  STAT
}

model User {
  id           String       @id @default(cuid())
  email        String       @unique
  name         String
  passwordHash String
  role         UserRole     @default(RADIOLOGIST)
  createdAt    DateTime     @default(now())
  updatedAt    DateTime     @updatedAt

  reports      Report[]     @relation("AssignedReports")
  auditLogs    AuditLog[]
}

model Report {
  id               String         @id @default(cuid())
  studyInstanceUID String
  status           ReportStatus   @default(DRAFT)
  body             String         @db.Text
  assignedTo       User           @relation("AssignedReports", fields: [assignedToId], references: [id])
  assignedToId     String
  templateId       String?
  createdAt        DateTime       @default(now())
  updatedAt        DateTime       @updatedAt

  versions         ReportVersion[]
  study            WorklistStudy  @relation(fields: [studyInstanceUID], references: [studyInstanceUID])
}

model WorklistStudy {
  id               String         @id @default(cuid())
  studyInstanceUID String         @unique
  accessionNumber  String
  patientName      String
  patientId        String
  modality         String
  studyDate        DateTime
  priority         StudyPriority  @default(ROUTINE)
  assignedToId     String?
  createdAt        DateTime       @default(now())
  updatedAt        DateTime       @updatedAt

  reports          Report[]
}
```

## Migrations

Migrations are managed exclusively with `prisma migrate`. Migration files are auto-generated and live in `prisma/migrations/`. Each migration directory contains a `migration.sql` file and is tracked in the `_prisma_migrations` table.

**Common commands:**

```bash
# Generate and apply a migration in development
npx prisma migrate dev --name describe_your_change

# Apply pending migrations in production
npx prisma migrate deploy

# Inspect current migration status
npx prisma migrate status
```

<Note>
  Never run raw SQL migration scripts directly against the production database. Always use `prisma migrate deploy` so that migration history remains consistent and Prisma can track applied migrations correctly. Ad-hoc SQL changes will cause schema drift and break future migrations.
</Note>

## Connection Pooling with PgBouncer

PgBouncer sits between the application and PostgreSQL in transaction pooling mode. This means each Prisma query acquires a connection from PgBouncer for the duration of the transaction, then returns it to the pool. This allows the application to maintain a large number of concurrent Prisma clients without exhausting PostgreSQL's `max_connections`.

Set your `DATABASE_URL` to point to PgBouncer, not directly to PostgreSQL:

```env
# .env
DATABASE_URL="postgresql://mosaic_app:password@pgbouncer:6432/mosaic?schema=public"
```

When using Prisma with PgBouncer, append `?pgbouncer=true` to the connection URL to disable prepared statement caching, which is incompatible with transaction pooling mode:

```env
DATABASE_URL="postgresql://mosaic_app:password@pgbouncer:6432/mosaic?schema=public&pgbouncer=true"
```
